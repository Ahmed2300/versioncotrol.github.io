المحاضرة الثانية: Git للعمل الجماعي

تلخيص وإعداد: أحمد عزام

تنبيه مهم: التعلم بالممارسة

لا تكتفِ بالقراءة فقط!

هذه المحاضرة مصممة للتطبيق العملي. افتح Terminal أو Command Prompt واكتب كل أمر بيديك. التعلم الحقيقي يحدث عندما تمارس الأوامر بنفسك وترى النتائج مباشرة.

نصيحة: جرب الأوامر، اكسر الأشياء، وتعلم من الأخطاء - هذه هي أفضل طريقة لإتقان Git!

شرح مرئي للمحاضرة

I. أساسيات العمل الجماعي: المستودعات البعيدة (Remotes)

ما هو الـ Remote؟

الـ Remote هو نسخة من مستودعك مستضافة على خادم (مثل GitHub أو GitLab). يسمح هذا لعدة مطورين بالعمل على نفس المشروع ومزامنة تغييراتهم. الاسم الافتراضي لأول remote هو origin.

git clone مقابل git init

  • git clone: تستخدمه عندما يكون المشروع موجودًا بالفعل على خادم بعيد، وتريد إنشاء نسخة محلية منه على جهازك.
  • git init: تستخدمه عندما تبدأ مشروعًا جديدًا من الصفر على جهازك المحلي.

المزامنة: Push, Pull, & Fetch

  • git push: يرفع (upload) تغييراتك المحلية إلى الـ remote.
  • git pull: يسحب (download) التغييرات من الـ remote ويدمجها فورًا.
  • git fetch: يسحب التغييرات من الـ remote لكن **لا** يدمجها، مما يسمح لك بمراجعتها أولاً.

II. رحلة عملية: من المحلي إلى العالمي

سنقوم بمحاكاة سير عمل كامل، من إنشاء مشروع محلي إلى رفعه والتعاون عليه.

الخطوة 0: الإعداد المسبق على GitHub

مطلوب قبل البدء

اذهب إلى حسابك في GitHub وأنشئ مستودعًا جديدًا وفارغًا (New Repository). **لا تقم بإضافة ملف README أو .gitignore من GitHub**. كل ما تحتاجه هو رابط الـ HTTPS الخاص بالمستودع، والذي سيبدو هكذا: https://github.com/YourUsername/ProjectName.git

الخطوة 1: إنشاء المشروع المحلي

إنشاء مجلد المشروع والانتقال إليه:

mkdir my-remote-project && cd my-remote-project

mkdir ينشئ مجلدًا جديدًا، و cd ينتقل إليه. الرمز && يعني "نفذ الأمر الثاني فقط إذا نجح الأول".

تهيئة مستودع Git محلي:

git init

ينشئ مستودع Git فارغ في المجلد الحالي. ستلاحظ ظهور مجلد مخفي .git يحتوي على كل بيانات Git.

فحص المستودعات البعيدة (قبل الربط):

git remote -v

يعرض قائمة بجميع المستودعات البعيدة المرتبطة بمشروعك. الآن ستكون القائمة فارغة لأننا لم نربط أي remote بعد. الخيار -v يعني "verbose" لإظهار التفاصيل الكاملة.

إنشاء ملف README وإضافة محتوى:

echo "# My Awesome Project" > README.md

echo يطبع النص، والرمز > يوجه الإخراج إلى ملف جديد (أو يستبدل محتوى ملف موجود).

إضافة الملف إلى منطقة التجهيز:

git add README.md

يضع الملف في منطقة التجهيز (Staging Area) ليصبح جاهزًا للـ commit.

إنشاء أول commit:

git commit -m "Initial commit"

يحفظ التغييرات بشكل دائم في تاريخ المستودع مع رسالة وصفية. الخيار -m يسمح بكتابة الرسالة مباشرة في سطر الأوامر.

الخطوة 2: ربط المحلي بالبعيد والدفع الأول

ربط المستودع المحلي بالمستودع البعيد:

git remote add origin https://github.com/YourUsername/ProjectName.git

يضيف مستودعًا بعيدًا جديدًا باسم `origin` ويربطه بالرابط المحدد. `origin` هو الاسم التقليدي للمستودع البعيد الرئيسي.

فحص المستودعات البعيدة (بعد الربط):

git remote -v

الآن ستظهر لك معلومات الـ remote المرتبط، مع عرض رابطين: أحدهما للـ `fetch` (السحب) والآخر للـ `push` (الرفع).

رفع الفرع المحلي إلى المستودع البعيد:

git push -u origin main

يرفع فرع `main` المحلي إلى الـ remote المسمى `origin`. الخيار `-u` (أو `--set-upstream`) يربط بين فرعك المحلي والفرع البعيد، لذا في المرات القادمة يمكنك فقط كتابة `git push` دون تحديد الـ remote والفرع.

ممتاز! اذهب الآن إلى صفحة مستودعك على GitHub، ستجد ملف README.md قد ظهر هناك.

الخطوة 3: محاكاة عمل زميل (استخدام Pull)

لنفترض أن زميلك في الفريق أضاف تعديلاً. سنحاكي ذلك عن طريق تعديل الملف مباشرة من واجهة GitHub.

  1. اذهب إلى مستودعك على GitHub.
  2. افتح ملف `README.md` واضغط على أيقونة القلم (Edit).
  3. أضف سطرًا جديدًا، ثم انزل للأسفل واضغط "Commit changes".

سحب التغييرات من المستودع البعيد:

git pull

يقوم بسحب آخر التغييرات من الـ remote ودمجها تلقائيًا في فرعك المحلي الحالي. هذا الأمر يجمع بين `git fetch` و `git merge`.

يمكنك استخدام `cat README.md` للتأكد من وجود السطر الجديد الذي أضافه "زميلك".

مشكلة محتملة: خطأ في التتبع

إذا ظهرت لك رسالة خطأ مثل:

There is no tracking information for the current branch.
Please specify which branch you want to merge with.
git pull <remote> <branch>

السبب: فرعك المحلي غير مرتبط بفرع بعيد (لم تستخدم `-u` في أول `push`).

الحلول:

  • الحل السريع: حدد الـ remote والفرع يدويًا:
    git pull origin main
  • الحل الدائم: اربط فرعك المحلي بالفرع البعيد:
    git branch --set-upstream-to=origin/main master
    ملاحظة: غيّر `master` إلى `main` إذا كان فرعك المحلي يسمى `main`

الخطوة 4: سير عمل الفروع (Feature Branch Workflow)

إنشاء فرع جديد للميزة والانتقال إليه:

git switch -c feature/add-contact-page

`git switch` هو الأمر الحديث للتنقل بين الفروع. الخيار `-c` ينشئ فرعًا جديدًا وينتقل إليه فورًا. هذا مكافئ لـ `git checkout -b`.

إنشاء ملف جديد للميزة:

echo "<h1>Contact Us</h1>" > contact.html

ينشئ ملف HTML جديد يحتوي على عنوان بسيط. هذا يمثل الميزة الجديدة التي نعمل عليها.

إضافة الملف الجديد إلى منطقة التجهيز:

git add contact.html

يضيف الملف الجديد إلى منطقة التجهيز ليصبح جاهزًا للـ commit.

حفظ التغييرات في commit:

git commit -m "feat: Add contact page"

يحفظ التغييرات مع رسالة وصفية تتبع نمط "Conventional Commits" حيث `feat:` يشير إلى ميزة جديدة.

الخطوة 5: دمج الميزة ورفع النتيجة النهائية

العودة إلى الفرع الرئيسي:

git switch main

ينتقل من فرع الميزة إلى الفرع الرئيسي `main` للاستعداد لعملية الدمج.

دمج فرع الميزة في الفرع الرئيسي:

git merge feature/add-contact-page

يدمج جميع التغييرات من فرع الميزة إلى الفرع الحالي (`main`). سينشئ Git commit دمج جديد إذا لزم الأمر.

رفع التغييرات إلى المستودع البعيد:

git push

يرفع الفرع الرئيسي المحدث إلى الـ remote. لا نحتاج لتحديد الـ remote والفرع لأننا استخدمنا `-u` في المرة الأولى.

عرض تاريخ المشروع بصريًا:

git log --oneline --graph --all

يعرض تاريخ جميع الفروع بشكل مختصر مع رسم بياني يوضح التفرع والدمج. `--oneline` يختصر كل commit لسطر واحد، `--graph` يرسم الخطوط، `--all` يعرض جميع الفروع.

مهمة متقدمة: إنشاء Pull Request كامل

سنتعلم كيفية دمج فرعين يحتويان على commits مختلفة وإنشاء Pull Request على GitHub.

الجزء الأول: إعداد المشروع (محلي وبعيد)

إنشاء مشروع جديد من الصفر:

mkdir my-git-project && cd my-git-project

تهيئة مستودع Git مع فرع main:

git init -b main

الخيار `-b main` ينشئ الفرع الرئيسي باسم `main` مباشرة (المعيار الحديث).

إنشاء أول ملف وcommit:

echo "# My Awesome Project" > README.md
git add README.md
git commit -m "Initial commit"
إنشاء مستودع على GitHub
  1. اذهب إلى github.com وسجل الدخول
  2. اضغط على "+" في الأعلى واختر "New repository"
  3. اكتب اسم المستودع (مثل: my-git-project)
  4. مهم: اتركه فارغًا - لا تضع README أو .gitignore
  5. اضغط "Create repository"

ربط المستودع المحلي بـ GitHub:

git remote add origin https://github.com/YourUsername/my-git-project.git

استبدل الرابط برابط مستودعك الفعلي من GitHub.

رفع أول commit:

git push -u origin main

الجزء الثاني: إنشاء فرع الميزة الجديدة

إنشاء والانتقال إلى فرع جديد:

git checkout -b add-user-feature

إضافة تغييرات في الفرع الجديد:

echo "User profile page." > user.html
git add user.html
git commit -m "Feat: Create user profile page"

الآن فرع `add-user-feature` يحتوي على commit جديد لا يوجد في `main`.

الجزء الثالث: إنشاء ودمج Pull Request

رفع فرع الميزة إلى GitHub:

git push origin add-user-feature
إنشاء Pull Request على GitHub
  1. اذهب إلى مستودعك على GitHub
  2. ستظهر رسالة صفراء مع زر "Compare & pull request" - اضغط عليه
  3. العنوان: "Add User Profile Feature"
  4. الوصف: اشرح التغييرات التي قمت بها
  5. تأكد أن Base branch هو `main` والـ Compare branch هو `add-user-feature`
  6. اضغط "Create pull request"
دمج Pull Request
  1. اضغط على الزر الأخضر "Merge pull request"
  2. اضغط "Confirm merge"
  3. مستحسن: احذف الفرع البعيد بالضغط على "Delete branch"

تهانينا! الكود من `add-user-feature` أصبح الآن جزءًا من `main` على GitHub.

الجزء الرابع: مزامنة وتنظيف المستودع المحلي

العودة إلى الفرع الرئيسي محليًا:

git checkout main

سحب آخر التغييرات من GitHub:

git pull origin main

هذا سيحمل merge commit الذي تم إنشاؤه على GitHub ويحديث فرعك المحلي.

حذف الفرع المحلي المكتمل:

git branch -d add-user-feature

بما أن العمل اكتمل ودُمج، لم نعد نحتاج هذا الفرع محليًا.

تم إكمال سير العمل!

أنت الآن على فرع `main` نظيف ومحدث، وقد أكملت دورة كاملة من إنشاء ميزة ودمجها عبر Pull Request.

III. أدوات متقدمة لتعديل التاريخ وسير العمل

git reset مقابل git revert (للتراجع عن الأخطاء)

قاعدة ذهبية: لا تستخدم `git reset` أبدًا على الـ commits التي تم رفعها إلى فرع مشترك.
  • git reset: يحذف الـ commits من التاريخ. استخدمه فقط محليًا لتصحيح أخطائك قبل عمل `push`.
  • git revert: ينشئ commit جديدًا يلغي تغييرات commit سابق. هذه هي الطريقة الآمنة والصحيحة للتراجع عن الأخطاء في الفروع المشتركة لأنها تحافظ على التاريخ.

العودة إلى commit معين باستخدام الـ Hash

أحيانًا تحتاج للعودة إلى نقطة معينة في تاريخ مشروعك. إليك الطرق المختلفة:

1. عرض تاريخ المشروع والحصول على الـ Hash

git log --oneline

يعرض قائمة مختصرة بجميع الـ commits مع الـ hash المختصر لكل منها (أول 7 أحرف).

2. العودة المؤقتة لمراجعة commit قديم

git checkout a1b2c3d

يضعك في حالة Detached HEAD لمراجعة الكود في هذه النقطة دون تعديل التاريخ. استبدل `a1b2c3d` بالـ hash الفعلي.

للعودة إلى الفرع الحالي:
git switch main

3. إلغاء commit معين (الطريقة الآمنة)

git revert a1b2c3d

ينشئ commit جديد يلغي تغييرات الـ commit المحدد. هذه هي الطريقة المفضلة في المشاريع المشتركة.

4. العودة الصعبة (خطير - للاستخدام المحلي فقط)

تحذير: هذا الأمر يحذف جميع الـ commits بعد النقطة المحددة. لا تستخدمه إذا كنت قد رفعت التغييرات إلى remote مشترك.
git reset --hard a1b2c3d

يعيد المشروع إلى الحالة الموجودة في الـ commit المحدد ويحذف كل شيء بعده نهائيًا.

git stash: الحافظة المؤقتة للمطورين

أداة قوية لحفظ التغييرات غير المكتملة مؤقتًا، مما يتيح لك التبديل بين المهام دون فقدان عملك.

متى تحتاج git stash؟

السيناريو الشائع: أنت تعمل على ميزة جديدة، لكن زميلك يطلب منك إصلاح خطأ عاجل في فرع آخر. تغييراتك الحالية غير جاهزة للـ commit، لكنك لا تريد فقدانها.

الأوامر الأساسية:

حفظ التغييرات الحالية:

git stash

أو مع رسالة وصفية:

git stash save "Working on user authentication"

عرض قائمة المحفوظات:

git stash list

يعرض قائمة مثل:

stash@{0}: On main: Working on user authentication
stash@{1}: WIP on main: 1a2b3c4 Add login form

استرجاع آخر محفوظة:

git stash pop

يطبق التغييرات ويحذف المحفوظة من القائمة.

تطبيق محفوظة دون حذفها:

git stash apply

أو تطبيق محفوظة معينة:

git stash apply stash@{1}

عرض محتوى محفوظة:

git stash show

أو لعرض تفصيلي:

git stash show -p

حذف محفوظة معينة:

git stash drop stash@{0}

حذف جميع المحفوظات:

git stash clear

مثال عملي كامل:

سير العمل:
  1. تعمل على ميزة جديدة وتعدل ملفات
  2. git stash save "Half-done feature X" - حفظ العمل
  3. git switch hotfix-branch - التبديل لإصلاح عاجل
  4. إصلاح الخطأ وعمل commit
  5. git switch main - العودة للفرع الأصلي
  6. git stash pop - استرجاع عملك السابق
  7. متابعة العمل على الميزة
ملاحظات مهمة:
  • git stash يحفظ فقط الملفات المتتبعة (tracked files)
  • الملفات الجديدة غير المضافة لن تُحفظ إلا إذا استخدمت git stash -u
  • استخدم git stash pop للاستخدام العادي
  • استخدم git stash apply إذا كنت تريد تطبيق نفس التغييرات في أماكن متعددة

أدوات مساعدة أخرى لسير العمل

  • git commit --amend: لتعديل آخر commit قمت به (إضافة تغييرات أو تغيير الرسالة) قبل عمل `push`.
  • git cherry-pick: لأخذ commit معين من فرع وتطبيقه على فرع آخر. مفيد لنقل إصلاح صغير دون دمج الفرع بأكمله.

دليل الأوامر الشامل

شرح مبسط لجميع أوامر Git الأساسية والمتقدمة

git checkout

يستخدم للتنقل بين أجزاء المستودع، بما في ذلك الانتقال إلى commit معين حتى لو لم يكن هناك فرع يشير إليه.

  • عند الانتقال لـ commit معين: يضعك في حالة "رأس منفصل" (detached head)
  • أي تغييرات ستفقد إلا إذا أنشأت فرعًا جديدًا
  • يمكن استهداف ملفات معينة مثل git reset

git switch

يستخدم للانتقال من فرع لآخر. أمر حديث ومحدد للتبديل بين الفروع.

  • أكثر وضوحًا من git checkout
  • مثال: git switch master للعودة للفرع الرئيسي

git log

يعرض تاريخ الـ commits بخيارات متعددة:

  • git log --oneline: عرض مختصر في سطر واحد
  • git log --author="Author Name": فلترة حسب المؤلف
  • git log --grep="Search Term": البحث في رسائل الـ commits
  • git log --graph: عرض بياني للفروع والدمج
  • للخروج: اضغط q

git init

ينشئ مستودع Git محلي جديد، ويخلق مجلد .git. يستخدم مرة واحدة عند بدء مشروع جديد.

git remote

يعرض معلومات المستودعات البعيدة:

  • git remote add [name] [URL]: إضافة مستودع بعيد
  • git remote -v: عرض روابط الـ fetch والـ push

git push

يرسل الـ commits المحلية إلى المستودع البعيد:

  • git push -u origin [branch name]: يربط الفرع المحلي بالبعيد
  • بعد الربط، يمكن استخدام git push فقط

git pull

يسحب التغييرات من المستودع البعيد ويدمجها في الفرع المحلي الحالي.

git clone

ينزل مستودعًا بعيدًا موجودًا إلى جهازك، مع كامل التاريخ والفروع وإعداد الـ remote. لا تحتاج git init أو git remote add بعده.

git fetch

ينزل البيانات من المستودع البعيد لكن لا يدمجها. يحديث فروع التتبع البعيدة فقط، مما يتيح لك رؤية التغييرات دون تطبيقها.

git status

يعرض حالة مجلد العمل ومنطقة التجهيز، بما في ذلك التغييرات المجهزة وغير المجهزة والملفات غير المتتبعة.

git add

يضع التغييرات من مجلد العمل إلى منطقة التجهيز للـ commit التالي.

git commit

يسجل التغييرات المجهزة في المستودع:

  • git commit -m "message": حفظ مع رسالة وصفية
  • git commit --amend: تعديل آخر commit
  • git commit --amend --no-edit: تعديل بدون تغيير الرسالة

git reset

يلغي التغييرات بنقل مؤشر HEAD لـ commit سابق:

  • --soft: يحتفظ بالتغييرات في منطقة التجهيز
  • --mixed (افتراضي): ينقل التغييرات لمجلد العمل
  • --hard: يحذف التغييرات نهائيًا (خطير!)

git revert

ينشئ commit جديد يلغي تغييرات commit محدد. آمن للفروع المشتركة لأنه يحافظ على التاريخ.

git cherry-pick

يطبق تغييرات commit معين من أي فرع على فرعك الحالي:

  • git cherry-pick --no-commit [hash]: يطبق التغييرات دون إنشاء commit

git stash

يحفظ التغييرات مؤقتًا للتبديل بين المهام:

  • git stash save "message": حفظ مع رسالة
  • git stash list: عرض قائمة المحفوظات
  • git stash apply: تطبيق دون حذف
  • git stash pop: تطبيق مع حذف
  • git stash drop: حذف محفوظة معينة

git reflog

يعرض سجل تحديثات HEAD والفروع في المستودع المحلي. مفيد لاسترداد الـ commits المفقودة حتى بعد hard reset.

IV. اختبار ومراجع

اختبر معلوماتك

git fetch يقوم بتنزيل البيانات الجديدة من الـ remote ولكنه لا يدمجها، بينما git pull يقوم بعمل fetch ثم merge لدمج التغييرات فورًا.

يُفضل revert في المستودعات المشتركة لأنه ينشئ commit جديدًا للتراجع بدلاً من حذف التاريخ، مما يحافظ على سلامة تاريخ المشروع لجميع المتعاونين.

يستخدم لحفظ التغييرات غير المكتملة بشكل مؤقت للسماح لك بالتبديل بين الفروع، ثم استرجاعها لاحقًا.
